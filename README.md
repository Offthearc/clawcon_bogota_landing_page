# Harness Engineering Template (React)

A template that configures Claude Code as a multi-agent pipeline
(`architect → designer → implementer → reviewer`) for building a **React +
TypeScript** front end one feature at a time, with state on disk and an
executable verification gate.

The application code under `src/` (beyond the scaffold) is generated by the
agents after you configure your project scope. Until you add features, the
suite is nearly empty — that is expected.

---

## What is the harness?

| Pillar                             | How it appears in this repo                                                                       |
| ---------------------------------- | ------------------------------------------------------------------------------------------------- |
| **1. The repo IS the system**      | `AGENTS.md`, `init.sh`, `feature_list.json`, `progress/`, `docs/`, `DESIGN.md`                    |
| **2. Multi-agent orchestration**   | `.claude/agents/leader.md`, `architect.md`, `designer.md`, `implementer.md`, `reviewer.md`        |
| **3. Supervision and improvement** | `CHECKPOINTS.md`, hooks in `.claude/settings.json`, the `scripts/design_check.mjs` gate, `tests/` |

---

## Stack

- **Vite + React 19 + TypeScript** (strict)
- **Vitest + React Testing Library** for unit/component tests
- **ESLint + Prettier** for lint/format, `tsc --noEmit` for types
- **npm** as the package manager
- **DESIGN.md** ([design.md format](https://github.com/google-labs-code/design.md))
  as the source of truth for the theme, with `src/theme/tokens.css` generated
  from it

## Using this template

### 1. Clone

```bash
git clone https://github.com/<your-org>/<your-project>.git
cd <your-project>
```

### 2. Install dependencies

```bash
npm install
```

### 3. Define your project scope

Edit `feature_list.json` with your project's features.
`feature_list.example.json` shows the format with a small React example.

```json
{
  "project": "my-project",
  "description": "Short description — the designer infers a theme from this.",
  "rules": {
    "one_feature_at_a_time": true,
    "require_tests_to_close": true,
    "valid_status": ["pending", "in_progress", "done", "blocked"]
  },
  "features": [
    {
      "id": 1,
      "name": "first_feature",
      "title": "First feature",
      "description": "Detailed description.",
      "acceptance": ["criterion 1", "criterion 2"],
      "status": "pending"
    }
  ]
}
```

> The `project` + `description` fields are what the `designer` agent reads to
> infer an appropriate theme into `DESIGN.md`.

### 4. Verify the harness

```bash
./init.sh
```

`init.sh` checks the toolchain, installs deps if needed, validates
`feature_list.json`, runs the **DESIGN.md conformance gate**, and runs the test
suite. Before the designer has tailored it, you will see
`[WARN] DESIGN.md not present yet` or the default starter theme — that is normal.

## Running it with Claude Code

Open Claude Code from the repo root:

```bash
claude
```

`CLAUDE.md` instructs the model to act as `leader` (orchestrate, never edit code
directly). Quick recipe:

1. `./init.sh` — must finish without errors in the toolchain/files/feature-list
   blocks (tests may be minimal while `src/` is still a scaffold).
2. Confirm `feature_list.json` has at least one feature with `status: "pending"`.
3. Ask Claude: **"implement the next pending feature"**.

What you will see in chat:

- The **leader** announces the plan, runs the **architect**, then the
  **designer** (which writes `DESIGN.md` + theme tokens), then an
  **implementer**, then a **reviewer**.
- **No code flows through chat** — only references like
  `done -> progress/impl_<feature>.md` (the anti-broken-telephone rule).

Where each subagent leaves its trace:

| File                           | Written by  | Contains                                                 |
| ------------------------------ | ----------- | -------------------------------------------------------- |
| `progress/current.md`          | leader      | Live plan for the session                                |
| `progress/arch_<…>.md`         | architect   | Architecture decisions                                   |
| `progress/design_<…>.md`       | designer    | Theme direction + token summary                          |
| `progress/impl_<feature>.md`   | implementer | Files touched + test output                              |
| `progress/review_<feature>.md` | reviewer    | Checklist against `docs/`, `DESIGN.md`, `CHECKPOINTS.md` |
| `feature_list.json`            | implementer | `pending` → `in_progress` → `done`                       |
| `progress/history.md`          | leader      | Append-only summary written at session close             |

## The DESIGN.md design gate

`DESIGN.md` holds machine-readable design tokens (YAML front matter) plus the
rationale (markdown). `scripts/design_check.mjs` does double duty:

- `npm run design:tokens` — generates `src/theme/tokens.css` from `DESIGN.md`
  (the designer runs this).
- `npm run design:check` — verifies the tokens file is in sync and that `src/`
  has no off-palette colors or off-token fonts. It is wired into `init.sh`, so
  the reviewer's "never approve with `./init.sh` red" rule enforces design
  conformance automatically. A PostToolUse hook also runs it for fast feedback,
  and the reviewer additionally checks each component against its `DESIGN.md`
  spec by eye.

## Repository structure

```
.
├── AGENTS.md                  # Navigation map for agents (progressive disclosure)
├── CHECKPOINTS.md             # Criteria for "correct final state" (C1–C6)
├── CLAUDE.md                  # Claude Code instructions (forces the leader role)
├── DESIGN.md                  # Design system: tokens + rationale (design.md format)
├── feature_list.json          # Active scope (one feature at a time)
├── feature_list.example.json  # Format reference (small React example)
├── init.sh                    # Environment verification + design/test gate
├── index.html                 # Vite entry
├── package.json               # Scripts + dependencies
├── tsconfig.json              # TypeScript config
├── vite.config.ts             # Vite + Vitest config
├── eslint.config.js           # ESLint flat config
├── progress/
│   ├── current.md             # Active session (live state)
│   └── history.md             # Append-only log
├── docs/
│   ├── architecture.md        # What "doing good work" means
│   ├── conventions.md         # Style, naming, design-token rules
│   └── verification.md        # How to prove work is correct
├── scripts/
│   ├── design_check.mjs       # Generates tokens.css + verifies DESIGN.md conformance
│   ├── validate_features.mjs  # feature_list.json validator (used by init.sh)
│   ├── detect_done.mjs        # Emits commit message when a feature moves to done
│   └── commit_on_done.sh      # Conventional-commit hook
├── .claude/
│   ├── agents/                # leader, architect, designer, implementer, reviewer
│   └── settings.json          # Hooks that automate verification
├── src/                       # React + TS app (src/theme/tokens.css is generated)
└── tests/                     # Vitest + React Testing Library tests
```

## Design principles illustrated by the harness

- **Progressive disclosure** in `AGENTS.md`: a map to find rules on demand.
- **One feature at a time**, enforced by `init.sh` (rejects more than one
  `in_progress` entry in `feature_list.json`).
- **State on disk, not in chat**: `progress/` survives restarts and blown
  context windows.
- **Executable verification**: `init.sh` runs real tests and the design gate —
  it does not trust what the agent says.
- **Design as an executable contract**: `DESIGN.md` drives the theme and
  `design_check.mjs` fails the build on drift.
- **Architect → Designer → Implementer → Reviewer**: the leader never
  implements, the implementer never self-approves, the reviewer never edits code.
- **Anti-broken-telephone**: subagents write results to files and return only
  a lightweight reference.
